2. TABLAS DE BASE DE DATOS (17 tablas)
================================================================================

2.1 availability_settings (Horarios de atencion)
-------------------------------------------------
- id                   uuid PK
- day_of_week          int (0=Domingo, 6=Sabado), UNIQUE
- start_time           time NOT NULL
- end_time             time NOT NULL
- slot_duration_minutes int DEFAULT 60
- is_active            boolean DEFAULT true
- created_at           timestamptz

2.2 blocked_dates (Fechas bloqueadas)
--------------------------------------

- date                 date UNIQUE NOT NULL

- created_at           timestamptz

2.3 bookings (Reservas)

9. FLUJO DE RESERVA (Maquina de estados)
================================================================================

servicios -> calendario -> formulario -> pago -> confirmacion

Paso 1: Servicios
- Lista de servicios activos de la tabla services
- Seleccionar uno (o saltar si hay servicio unico)

Paso 2: Calendario (Calendar.tsx)
- Navegacion por mes (anterior/siguiente)
- Dias disponibles resaltados (de availability_settings donde is_active=true)
- Fechas bloqueadas en gris (de blocked_dates)
- Generacion de slots: start_time -> end_time, intervalo = slot_duration_minutes
- Verificacion de capacidad: contar reservas existentes por slot
- Formulario de lista de espera cuando el slot esta lleno

Paso 3: Formulario (BookingForm.tsx)
- Nombre: texto, validacion minimo 3 caracteres
- Telefono: texto, regex /^\+?[\d\s-]{8,}$/
- Email: texto, regex /^[^\s@]+@[^\s@]+\.[^\s@]+$/
- Verificacion de duplicados: previene mismo email con reserva pendiente en misma fecha/hora

Paso 4: Pago (Payment.tsx)
- Integracion con MercadoPago Brick (formulario de tarjeta)
- Muestra monto + moneda
- Polling para verificar estado de aprobacion

Paso 5: Confirmacion (Confirmation.tsx)
- Muestra codigo de reserva (RES-YYYY-XXXX)
- Boton de copiar al portapapeles
- Dispara edge function send-confirmation-email
- Resumen de detalles de la reserva

12. SEGURIDAD
================================================================================

- RLS (Row-Level Security) habilitado en todas las tablas
- Autenticacion admin: email+password via RPC (verify_admin_password)
- Credenciales almacenadas en sessionStorage
- Todas las funciones admin autentican via headers email+password
- Lectura publica: availability_settings, blocked_dates, settings, services
- Insercion anonima: bookings (el publico puede crear)
- CRUD admin: todas las operaciones via edge functions con service role
- Sin secretos en frontend: public token de MP se obtiene via get-mp-config
